## Principles

1. **Additive, non-destructive — 기본 동작/출력 불변.** goksung은 기존 흐름 위의 조건부 분기. `description`(파일 메타데이터) 편집은 항상 존재하지만, **키워드가 없으면 기본 모드 출력/동작은 오늘과 완전히 동일**해야 한다.
2. **Grader-legible, reader-natural.** 채점자가 5개 카테고리를 각 섹션에서 즉시 식별할 수 있어야 하되, 일반 독자에게는 여전히 자연스러운 커뮤니티 사례글로 읽혀야 한다.
3. **Weight-proportional effort.** 배점이 큰 두 카테고리(25점씩)에만 엄격한 콘텐츠 게이트, 20점 쌍은 표준 수준, 10점 항목은 폴리시 전용.
4. **인터랙티브 계약 유지.** Phase 3은 한 번에 하나씩 질문하며, todo/task 목록에 등록하지 않는다. 새로 추가되는 채점 자가진단 단계도 이 가드에 명시적으로 포함된다.
5. **Never fabricate.** 근거가 없으면 "해당 없음"으로 정직하게 표시하고 통과시킨다 — 절대 지어내지 않는다.
6. **Tool-agnostic.** goksung 로직은 DEVLOG.md와 사용자 답변만 참조하며 특정 도구 경로에 의존하지 않는다 (Claude Code/OpenCode/Codex/Gemini/Antigravity 공통).

## Acceptance Criteria (10개, 테스트 가능)

1. **기본 no-op:** goksung 키워드 없는 실행에서 goksung 전용 질문/헤딩/부록/게이트가 전혀 출력되지 않는다. 모든 goksung 블록이 `IF goksung` 조건 + 편집2의 no-op 문구로 감싸져 있는지 확인.
2. **단일 키워드 트리거:** 리터럴 "goksung" 존재 시에만 활성화, 부재 시 goksung 동작 전무 (2차 질문 분기 없음).
3. **카테고리 매핑:** goksung 출력에서 5개 루브릭 카테고리가 각각 식별 가능한 섹션에 매핑됨.
4. **25점 HARD 게이트:** 정량 before/after 수치 ≥1 AND (스크린샷 추천 OR 결과 링크) ≥1 AND 재사용 프롬프트 ≥2 (결과물완성도), AND cross-domain 시나리오 ≥1 (업무적용) — 또는 기준 8의 정직한 "해당 없음".
5. **20점 표준:** 🔧작업과정 에피소드 ≥2(각 사용자 원문 인용); 문제상황에 정량화된 불편함 + 계기 명시.
6. **독자 자연스러움:** 본문 헤딩에 점수 라벨 없음; 점수 매핑은 선택적 삭제 가능 부록에만 존재.
7. **계약 유지:** Phase 3 단계 및 채점 자가진단이 모두 todo 목록에서 제외됨(편집4 반영 확인); 본문에 소스코드/명령어 없음; 5개 도구 전반에서 동일 동작.
8. **예외 처리(3단계):** 게이트 미충족 시 (1) 근거 재확인 → (2) 사용자에게 1회 질문(단, 2단계 중복 질문 방지) → (3) 그래도 없으면 정직하게 "해당 없음" 표시 후 통과 — 절대 지어내지 않음.
9. **근거 조작 없음:** DEVLOG/답변에 근거 없는 지표·스크린샷·프롬프트·시나리오가 출력에 등장하지 않음.

## ADR

- **Decision:** `write-post` 스킬에 키워드 트리거 "goksung" 모드를 5개 additive 편집으로 추가. 실행 모드 블록(키워드 전용, 침묵 기본값), Phase 3 goksung 전용 후속질문(루브릭 카테고리 매핑), 5단계 머리에 폴딩된 자가진단 게이트(3단계 anti-fabrication 예외 처리 내장), 385줄 todo-가드에 채점 자가진단 명시적 추가, 기본 OFF·삭제 가능한 채점 대응표 부록. 설치 위치: `/Users/tearess-mac/gpters/write-post-goksung/.claude/skills/write-post/SKILL.md`.
- **Drivers:** (1) 고정 100점 루브릭 하 점수 최대화 (2) 기본 모드 무회귀 + 5개 도구 범용성 (3) 사용자의 로컬 실사용 의도 부합.
- **Alternatives considered:** 모호 시 질문하는 트리거(기각 — 기본 침묵 원칙 위반) / 독립 Phase 4(기각 — Phase 3 전용 todo 가드를 우회) / 본문 내 점수 라벨(기각 — rubric-gaming처럼 보여 사례공유력 저해) / 채점 대응표 기본 ON(기각 — 동일한 신뢰성 이슈) / upstream PR 우선(연기 — 사용자의 명시적 의도와 불일치, fast-follow로 보존).
- **Why chosen:** 루브릭을 만족시키면서 기본 침묵과 정직한(비조작) 근거 처리를 보장하는 가장 작은 additive 변경; 가중 게이트가 요구하는 콘텐츠는 기존 템플릿 섹션(531/537/550/557줄)에 이미 자리가 있어 실현 가능성 확인됨.
- **Consequences:** SKILL.md가 약 40줄 증가; 유지보수 대상으로 조건부 분기 2개 + 게이트 루프 1개 추가.
- **Follow-ups:** (1) 동일 delta로 `daht-mad/write-post`에 upstream PR (2) 검증 절차 (a)-(d)로 실제 실행 검증 (3) README에 goksung 옵션 문서화는 upstream PR 시점에 진행.
- **Residual risk (accepted):** 가중 게이트는 5개 중 3개 카테고리에서 본질적으로 "콘텐츠 존재 여부" 문제이며, 이론적으로 template-stuffing처럼 보일 위험이 있다. **완화:** 대상 섹션이 이미 템플릿에 실재하고(531/537/550/557줄), anti-fabrication 3단계 + 정직한 "해당 없음" 설계로 무결성 리스크를 낮춤. 잔여 위험은 낮음으로 판단, 수용.
