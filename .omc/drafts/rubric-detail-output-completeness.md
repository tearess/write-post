# 결과물 완성도 (Output Completeness) — 25점 세부 채점 항목 (그물식)

> 대상: write-post 스킬이 생성하는 `AI_CASE_STUDY.md`
> 채점 방식: 사람 채점자가 100점 루브릭 중 이 카테고리(25점, 최대 가중치 동률)를 채점
> 설계 원칙: "실제로 동작한다는 증거 > 주장", "정량 수치 > 형용사", "재현/재사용 가능성 > 일회성". HARD 게이트 카테고리이므로 충족기준은 최대한 이분법적으로 확인 가능하게 서술.
> 관련 템플릿 섹션: `✅ 결과 (After)`(Before/After 지표표 + `결과물` 하위섹션), `📋 재사용 가능한 프롬프트`
> 관련 Phase-3 질문: "결과 임팩트: 결과물로 인해 구체적으로 뭐가 달라졌나요? (시간 절약, 효율 등)" + goksung 후속 "Before→After를 정량 수치 하나로. 결과물 스크린샷/링크 있나요?" / "핵심 팁"의 goksung 후속 "재사용 프롬프트 2개".

---

## 세부 채점 항목 (10개)

### 1. 동작 증거의 존재 (Proof-of-Work)
- **왜 채점자가 이걸 볼 것 같은지**: "완성도"의 1차 관문은 "이게 진짜 돌아가긴 하는가"이다. 채점자는 글만 그럴듯하고 증거가 하나도 없는 "썰(narrative-only)" 케이스를 가장 먼저 걸러낸다. 결과물 완성도 점수의 바닥을 결정하는 항목.
- **충족기준**:
  - 충분함: 결과물이 실재함을 제3자가 확인 가능한 증거가 최소 1개 존재 — 스크린샷(UI/출력 화면), 접속 가능한 링크(배포 URL·레포·데모), 실행 로그/터미널 출력, 또는 산출물 파일 캡처 중 하나 이상.
  - 부족함: "완성했다/잘 됐다"는 서술은 있으나 증거가 텍스트 주장뿐이고, 스크린샷·링크·로그가 전부 없음.
  - 없음: 결과물이 동작했다는 언급 자체가 없거나, "언젠가 만들 예정"류의 미래형만 존재.
- **write-post 연결점**: `✅ 결과 (After) > 결과물` 하위섹션(스크린샷/링크)에 직접 매핑. goksung 후속 "결과물 스크린샷/링크 있나요?"가 이미 커버. **HARD 게이트 후보(A).**

### 2. 정량 Before/After 지표 (Quantified Delta)
- **왜 채점자가 이걸 볼 것 같은지**: 임팩트를 형용사("훨씬 빨라짐")가 아니라 숫자로 말할 수 있는가가 완성도 상위권을 가른다. 채점자는 "3시간 → 20분", "수작업 50건/일 → 자동 500건/일" 같은 한 줄을 찾는다.
- **충족기준**:
  - 충분함: Before 값과 After 값이 **둘 다 수치**로 제시되고 단위가 명확한 지표가 최소 1개(시간/비용/처리량/오류율/전환율 등). 변화 방향과 크기가 계산 가능.
  - 부족함: After 수치만 있고 Before 기준선이 없음(비교 불가), 또는 "약 절반", "많이"처럼 정량화되지 않은 표현만.
  - 없음: 임팩트에 대한 수치가 전무하고 정성 서술만.
- **write-post 연결점**: `✅ 결과 (After)`의 Before/After 지표표에 매핑. goksung 후속 "Before→After를 정량 수치 하나로"가 이미 커버. **HARD 게이트 후보(A).**

### 3. 엔드투엔드 완결성 (End-to-End Completeness)
- **왜 채점자가 이걸 볼 것 같은지**: 부분 구현(핵심 로직만 있고 실제로 이어 붙이면 안 돌아감) vs 입력→처리→출력이 한 흐름으로 완결된 산출물은 완성도에서 체감 차이가 크다. 채점자는 "데모를 처음부터 끝까지 돌릴 수 있는가"를 상상한다.
- **충족기준**:
  - 충분함: 사용자가 실제로 사용하는 전체 플로우(진입→핵심동작→결과획득)가 작동함이 서술/증거로 드러남. 남은 부분이 있다면 "무엇이 미완인지" 명시.
  - 부족함: 일부 단계만 구현되어 있고 연결부(배포/연동/인증 등)가 비어 있으나 그 사실이 흐릿하게 처리됨.
  - 없음: 파편적 조각(스니펫·목업)만 있고 하나의 사용 흐름으로 이어지지 않음.
- **write-post 연결점**: 본문 구현 서술 + `결과물` 섹션 종합 판단. 전용 질문 없음 → **신규 후속 후보**: "이 결과물, 지금 처음부터 끝까지 한 번에 돌아가나요? 아직 빠진 연결 단계가 있다면 무엇인가요?" (단, A그룹 과밀 방지를 위해 선택적 심화로 둘 수 있음).

### 4. 예외/엣지케이스 처리 (Robustness Beyond Happy Path)
- **왜 채점자가 이걸 볼 것 같은지**: "데모에서만 되는" 취약한 happy-path 빌드와 에러 처리·예외 입력을 견디는 결과물은 완성도 격차가 크다. 채점자는 "이상한 입력 넣으면 터지나?"를 본다.
- **충족기준**:
  - 충분함: 최소 1개의 실패/예외 상황(빈 입력, 오류 응답, 한도 초과, 잘못된 형식 등)을 어떻게 다루는지 구체적으로 서술.
  - 부족함: 정상 경로만 언급하고 예외는 "추후 보완"으로 미룸.
  - 없음: 견고성에 대한 고려가 전혀 드러나지 않음(happy-path만).
- **write-post 연결점**: 본문 구현/트러블슈팅 서술에 산발적으로 존재. 전용 질문 없음 → **선택적 심화(B)**: 별도 필수 질문화하면 인터뷰 마찰이 큼. 있으면 가점, 없어도 감점 최소.

### 5. 시각 증거의 품질·배치 적절성 (Visual Evidence Quality & Placement)
- **왜 채점자가 이걸 볼 것 같은지**: 스크린샷이 있어도 흐릿하거나 맥락 없이 아무 데나 붙어 있으면 설득력이 떨어진다. 채점자는 "이 이미지가 무엇을 증명하는지 한눈에 오는가"를 본다.
- **충족기준**:
  - 충분함: 시각 자료가 (a) 결과/임팩트를 직접 보여주는 장면이고 (b) 해당 주장 문단 근처에 배치되며 (c) 캡션/설명으로 "무엇을 보는지" 안내됨.
  - 부족함: 이미지가 있으나 캡션·맥락 없이 나열되거나, 결과와 무관한 장식성 이미지.
  - 없음: 시각 자료 없음(항목 1과 별개로, 여기서는 "품질" 축).
- **write-post 연결점**: `결과물` 섹션. 항목 1이 존재 여부(HARD)를, 이 항목은 품질을 담당 → **선택적 심화(B)**. 스킬이 스크린샷 배치·캡션 가이드를 템플릿 힌트로 제공하면 충분(질문 추가 불필요).

### 6. 결과의 객관적 검증가능성 (Objective Verifiability)
- **왜 채점자가 이걸 볼 것 같은지**: 순수 자기평가("제 생각엔 훌륭합니다")와 제3자가 직접 확인·재현할 수 있는 증거는 신뢰도가 다르다. 채점자는 "내가 이 링크를 눌러/이 수치를 되짚어 검증할 수 있나"를 본다.
- **충족기준**:
  - 충분함: 지표의 출처·측정 방법이 드러나거나(어떻게 3시간→20분을 쟀는지), 결과를 남이 직접 확인할 수 있는 경로(라이브 링크·공개 레포·실행법)가 제공됨.
  - 부족함: 수치는 있으나 측정 근거가 불명확하고 검증 경로도 비공개.
  - 없음: 전적으로 주관적 자평만 있고 어떤 검증 수단도 없음.
- **write-post 연결점**: `결과 (After)` 지표표 + `결과물` 링크. 항목 1·2와 강한 상관 → 별도 필수 질문 대신, 항목 2 goksung 후속에 "수치를 어떻게 측정했는지 한 줄"을 옵션으로 얹는 정도. 기본 **선택적 심화(B)**.

### 7. 재사용 프롬프트의 실질성 (Reusable Prompt Substance)
- **왜 채점자가 이걸 볼 것 같은지**: GPTers 글의 핵심 가치는 "독자가 그대로 따라 쓸 수 있는가"이다. 채점자는 재사용 프롬프트가 실제로 복붙해서 쓸 수 있는 완결형인지, 아니면 "프롬프트를 잘 쓰세요" 수준의 공허한 팁인지 구분한다.
- **충족기준**:
  - 충분함: 복사해서 바로 쓸 수 있는 프롬프트가 **2개 이상**, 각각 목적/맥락(언제 쓰는지)이 붙고 플레이스홀더 등 사용법이 명확.
  - 부족함: 프롬프트가 1개뿐이거나, 2개여도 너무 추상적("코드를 잘 짜줘")이라 재사용 가치가 낮음.
  - 없음: 재사용 프롬프트 섹션이 비어 있거나 일반 조언만.
- **write-post 연결점**: `📋 재사용 가능한 프롬프트` 섹션 + "핵심 팁" goksung 후속(재사용 프롬프트 2개)이 이미 커버. **HARD 게이트 후보(A).**

### 8. 결과의 지속 재현성 (Durable Reusability, Not One-Off)
- **왜 채점자가 이걸 볼 것 같은지**: 오늘만 되고 내일 깨지는 일회성 결과물(하드코딩·수동 세팅 의존)과 반복 실행해도 같은 결과가 나오는 산출물은 완성도가 다르다. 채점자는 "일주일 뒤에도 이게 동작할까"를 본다.
- **충족기준**:
  - 충분함: 결과물이 재현 가능한 형태(스크립트/템플릿/배포물 등)이고, 반복 사용·다른 입력에도 동작함이 드러남. 또는 재현에 필요한 준비물이 명시됨.
  - 부족함: 특정 데이터·수동 조작에 강하게 의존해 재실행 보장이 약함.
  - 없음: 명백히 일회성(수동 결과 한 번, 재현 수단 없음).
- **write-post 연결점**: 본문 + `재사용 프롬프트`가 부분적으로 지지. 전용 질문 없음 → **선택적 심화(B)**. 항목 3(완결성)·7(재사용 프롬프트)과 묶어 판단.

### 9. 미완/한계의 정직한 명시 (Honest Scope of Completion)
- **왜 채점자가 이걸 볼 것 같은지**: 완성도를 과장하는 글은 채점자가 곧 간파하고 신뢰도 전체를 깎는다. 반대로 "여기까지 됐고, 이건 아직 미완"을 정직히 밝히면 완성도 판단이 정확해지고 오히려 신뢰가 오른다. anti-fabrication 원칙과 직결.
- **충족기준**:
  - 충분함: 무엇이 완성됐고 무엇이 미완/한계인지 경계가 명확. 증거 없는 항목은 "해당 없음"으로 정직히 표기.
  - 부족함: 전부 "완성"으로 뭉뚱그리고 한계 언급이 없어 실제 완성도 파악이 어려움.
  - 없음: 검증 불가한 과장/허위 정황(있지도 않은 지표·링크 암시).
- **write-post 연결점**: 전 섹션 공통 원칙. 스킬의 anti-fabrication 가드(증거 없으면 "해당 없음" 기록)로 강제 → **HARD 게이트의 escape hatch 규칙**으로 편입. 별도 질문보다 생성 규칙으로 처리.

### 10. 결과-목표 정합성 (Result Matches the Stated Goal)
- **왜 채점자가 이걸 볼 것 같은지**: 결과물이 훌륭해도 애초 하려던 것과 어긋나면 "완성"이라 부르기 어렵다. 채점자는 도입부의 문제/목표와 결과가 같은 것을 가리키는지 대조한다.
- **충족기준**:
  - 충분함: 서두에 제시한 문제/목표를 결과물이 직접 해결했음이 대응되게 드러남(목표 지표 = After 지표).
  - 부족함: 결과는 있으나 처음 목표와 초점이 어긋나거나, 목표 대비 일부만 해결하고 그 사실이 불명확.
  - 없음: 목표와 결과가 서로 무관하거나 목표 자체가 흐릿.
- **write-post 연결점**: 도입/문제 섹션 ↔ `결과 (After)` 대조. 전용 질문 없음 → **선택적 심화(B)**. 생성 시 정합성 체크로 처리 가능(질문 추가 불필요).

## (A) 필수 확인 질문 / HARD 게이트 후보

> 25점·HARD 게이트 카테고리이므로 20점급보다 게이트를 약간 넓게 잡되, 최대 마찰을 피하기 위해 **핵심 3개 + 규칙 1개**로 압축. 각 게이트에는 anti-fabrication escape hatch 적용: 진짜로 없으면 지어내지 말고 **"해당 없음"으로 정직히 기록**하고 통과(단, 그 사실이 완성도 서술에 반영되어야 함).

1. **동작 증거 존재 (항목 1)** — 스크린샷 / 접속 링크 / 실행 로그 중 최소 1개.
   - 게이트: 하나도 없으면 goksung 후속 재질문 → 그래도 없으면 `결과물: 해당 없음(증거 미확보)` 명기 후 통과. (증거 없는 완성 주장은 감점 신호로 남김.)
2. **정량 Before/After 지표 (항목 2)** — Before·After 둘 다 수치 1쌍 이상.
   - 게이트: 정량 수치 부재 시 "Before→After를 수치 하나로" 재질문 → 정말 없으면 정성 서술로 대체하되 "정량 미측정" 명기.
3. **재사용 프롬프트 2개 이상 (항목 7)** — 복붙 가능·목적 명시.
   - 게이트: 2개 미만이면 "핵심 팁" goksung 후속으로 보강 요청 → 확보 실패 시 확보된 개수만 정직히 게재.
4. **정직성 규칙 (항목 9, escape hatch)** — 위 1~3에서 확보 실패 시 반드시 "해당 없음"으로 표기하고 절대 날조 금지. 이는 질문이 아니라 **전 생성 과정에 적용되는 HARD 규칙**.

## (B) 선택적 심화 확인 (가점용, 별도 질문 없이 생성 규칙·템플릿 힌트로 처리)

- **항목 3 엔드투엔드 완결성** — 흐름 완결 여부. 미완이면 무엇이 빠졌는지 한 줄만 유도(강제 질문화하지 않음).
- **항목 4 예외/엣지케이스 처리** — 있으면 가점, 본문 트러블슈팅에서 자연 포착.
- **항목 5 시각 증거 품질·배치** — 캡션/맥락 가이드를 템플릿 힌트로 제공.
- **항목 6 객관적 검증가능성** — 측정 방법 한 줄을 항목 2 후속에 옵션으로 얹기.
- **항목 8 지속 재현성** — 재사용 프롬프트·산출물 형태로 간접 판단.
- **항목 10 결과-목표 정합성** — 생성 시 도입부↔결과 대조 자동 체크.

### 설계 메모
- A그룹은 기존 write-post 흐름(`결과물` 링크/스크린샷, `Before/After` 지표표, `재사용 프롬프트` 2개)에 **이미 시딩된 3개 질문과 1:1로 정렬** → 신규 질문 추가 없이 게이트화 가능. 인터뷰 마찰 증가 0.
- B그룹은 전부 "질문"이 아니라 **생성 규칙·템플릿 힌트**로 흡수 → 완성도 상한(가점)은 열어두되 사용자 부담은 최소화.
- 신규 후속이 정말 필요한 유일 후보는 항목 3(완결성) "지금 처음부터 끝까지 돌아가나요?"이지만, A 과밀 방지를 위해 기본은 B로 배치하고 팀 판단에 위임.
