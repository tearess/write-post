# 업무 적용 가능성 (25점) — 세부 채점 항목 그물식 설계

> 대상 카테고리: **업무 적용 가능성 / Work·Business Applicability (25점, 최대 배점 공동 1위)**
> 대응 템플릿 섹션: `🌍 다른 업무에 적용한다면?`(550줄), `📋 재사용 가능한 프롬프트`(557줄)
> 현행 HARD 게이트: 구체적 cross-domain 시나리오 ≥1 (막연한 "다른 데도 쓸 수 있음" 불가)
> 대응 Phase-3 goksung 후속질문: Q7(핵심 팁→재사용 프롬프트 2개), Q8(향후 계획→전혀 다른 업무 예시 1개)
>
> 설계 원칙: 25점 HARD 게이트 카테고리이므로 각 충족기준은 **충분함 / 부족함 / 없음** 3단으로 체크 가능하게 정의한다. anti-fabrication 원칙상, 근거 없는 시나리오·프롬프트는 "지어내기"이며 "해당 없음" 표기가 정답이다.

---

## 세부 채점 항목 (11개)

### 1. 진짜 cross-domain 전이성 (핵심)
- **항목명**: 전혀 다른 직무/도메인으로의 실질적 전이 예시
- **왜 채점자가 이걸 볼 것 같은지**: "업무 적용 가능성"의 본질은 '이 사람이 한 것'이 아니라 '이 방식을 남이/다른 데서 쓸 수 있는가'다. 채점자는 저자 본인 업무의 연장(같은 직무 내 반복)인지, 아니면 마케터→인사, 개발→회계처럼 **직무 경계를 넘는** 이식인지를 가장 먼저 가른다.
- **충족기준**:
  - 충분함: 저자 원래 도메인과 명확히 다른 직무/부서/업종 1개 이상을 지목하고, "무엇을(대상 업무) + 어떻게(같은 방식의 어느 부분을) + 왜 될 것 같은지(공통 구조)"가 한 문단 안에 다 있음.
  - 부족함: 다른 업무를 언급하되 "마케팅에도 활용 가능"처럼 대상만 있고 방법·근거가 없음.
  - 없음: "여러 분야에 두루 쓸 수 있다" 같은 도메인 미지정 일반론뿐.
- **write-post 연결점**: `🌍 다른 업무에 적용한다면?` + Q8 goksung 후속("전혀 다른 업무에 쓴다면 구체적으로 뭘? 예시 1개"). **이미 커버됨 (현행 HARD 게이트 본체).**

### 2. 재사용 가능한 산출물의 존재
- **항목명**: 복붙 가능한 딜리버러블(프롬프트/템플릿/워크플로우)
- **왜 채점자가 이걸 볼 것 같은지**: 적용 가능성은 '독자가 지금 가져다 쓸 물건'이 있을 때 급등한다. 영감만 주는 글과, 프롬프트 블록을 그대로 복사해 갈 수 있는 글은 등급이 다르다.
- **충족기준**:
  - 충분함: 코드블록/인용블록 형태로 **그대로 복사 가능한** 프롬프트 또는 템플릿이 ≥2개, 각각 무엇을 하는 것인지 한 줄 설명 포함.
  - 부족함: 재사용물이 1개거나, 산문 속에 녹아 있어 경계가 불명확(복붙 단위가 안 됨).
  - 없음: "이런 식으로 물어보면 된다"는 서술만 있고 복사 가능한 실체 없음.
- **write-post 연결점**: `📋 재사용 가능한 프롬프트` + Q7 goksung 후속("바로 복붙할 재사용 프롬프트 2개"). **이미 커버됨 (게이트: 재사용 프롬프트 ≥2).**

### 3. 재사용 산출물의 파라미터화(범용성)
- **항목명**: 딜리버러블이 '내 케이스 전용'이 아니라 변수화되어 있는가
- **왜 채점자가 이걸 볼 것 같은지**: 복붙 가능해도 저자의 회사명·특정 파일명·1회성 맥락이 하드코딩돼 있으면 남이 못 쓴다. 채점자는 프롬프트에 `[변수]` 자리표시나 "이 부분을 본인 상황으로 바꾸세요" 안내가 있는지를 본다.
- **충족기준**:
  - 충분함: 재사용 프롬프트에 치환 지점(플레이스홀더/변수/괄호 표기)이 명시돼 독자가 자기 맥락으로 즉시 교체 가능.
  - 부족함: 일부만 범용이고 저자 고유 맥락이 섞여 있어 부분 수정 필요.
  - 없음: 저자 상황에 완전히 고정돼 그대로는 재사용 불가.
- **write-post 연결점**: `📋 재사용 가능한 프롬프트`. **부분 커버 — 신규 후속질문 후보**: Q7에 "이 프롬프트에서 남이 바꿔 써야 할 부분을 `[대괄호]`로 표시할까요?" 미세 추가 권장(게이트화까지는 불필요).

### 4. 즉시 실행 가능한 구체성 수준
- **항목명**: 유사 문제를 가진 독자가 '바로 따라 할' 수 있는 해상도
- **왜 채점자가 이걸 볼 것 같은지**: 적용 가능성의 실전 지표는 "읽고 나면 내가 뭘 해야 할지 아는가"다. 채점자는 추상적 영감(막연)과 실행 레시피(구체) 사이 어디쯤인지를 판정한다.
- **충족기준**:
  - 충분함: 적용 시나리오 또는 팁이 '무엇을→어떤 도구/프롬프트로→어떤 순서로'까지 내려가 독자가 다음 행동을 특정할 수 있음.
  - 부족함: 방향은 주되 첫 단계가 모호("AI에게 잘 시키면 된다" 수준).
  - 없음: 감상·동기부여만 있고 행동 지침 없음.
- **write-post 연결점**: `🌍 다른 업무에 적용한다면?` + `📋 재사용 가능한 프롬프트`. **이미 커버됨** (게이트의 "막연한 표현 불가" 조건이 이 축을 방어).

### 5. 비개발자 실행 장벽(로우코드/노코드 접근성)
- **항목명**: 개발 지식 없이도 재현 가능한가
- **왜 채점자가 이걸 볼 것 같은지**: GPTers 독자층은 상당수가 비개발 직무다. 채점자는 "이 방식을 재현하려면 코딩/터미널/환경설정이 필수인가, 아니면 대화형 프롬프트만으로 되는가"를 적용 가능성의 저변 폭으로 본다. (본문 게이트가 소스코드 노출을 막는 것과도 정합.)
- **충족기준**:
  - 충분함: 재현에 필요한 것이 대화형 AI + 복붙 프롬프트 수준이거나, 개발 요소가 있어도 "비개발자는 이 부분만 하면 된다"는 우회 경로가 명시됨.
  - 부족함: 진입장벽에 대한 언급 없이 기술 전제(특정 CLI/코드)가 암묵적으로 깔림.
  - 없음: 개발 지식·환경이 사실상 필수인데 그 사실조차 안내 안 됨.
- **write-post 연결점**: `🌍 다른 업무에 적용한다면?` + 본문 "소스코드 없음" 게이트(기준7). **부분 커버 — 선택적 심화**: 필요 시 팁 섹션에 "비개발자가 따라 할 때 주의점" 한 줄 유도.

### 6. ROI 프레이밍의 설득력·일반화 가능성
- **항목명**: 주장된 시간/비용 절감이 개연성 있고 남에게도 재현되는가
- **왜 채점자가 이걸 볼 것 같은지**: "3시간→10분" 같은 수치가 일회성 운(fluke)인지, 반복·타인에게도 성립하는 구조적 이득인지를 채점자는 의심한다. 적용 가능성 점수는 ROI가 **일반화 가능**하다고 믿길 때만 온전히 부여된다.
- **충족기준**:
  - 충분함: 절감 효과가 반복 가능한 이유(자동화되는 반복 작업, 재사용 자산 등)와 함께 제시돼, 다른 사람/다음 번에도 유사 효과가 날 것이라 납득됨.
  - 부족함: 수치는 있으나 "이번엔 운 좋게" 식 1회성이거나 조건이 특이해 일반화 근거가 약함.
  - 없음: 정성적 "많이 편해졌다"만 있고 재현 논리 없음. (단, 근거 없는 수치 날조는 오히려 감점 — anti-fabrication.)
- **write-post 연결점**: `🌍 다른 업무에 적용한다면?` (결과물완성도 카테고리의 정량 before/after와 인접하나, 여기선 '그 이득의 이식 가능성'이 초점). **부분 커버 — 신규 후속질문 후보(선택)**: "이 절감이 다음에도/남에게도 재현될 이유는?" 1회 질문.

### 7. 적용 범위의 폭(1개 vs 다수 도메인)
- **항목명**: 인접 1개 유스케이스인가, 서로 다른 복수 도메인인가
- **왜 채점자가 이걸 볼 것 같은지**: 게이트는 ≥1이지만, 채점자는 25점 만점을 향해 '얼마나 넓게 뻗는가'로 가점한다. 성격이 다른 도메인 2~3개로 뻗으면 방법의 범용성이 입증된다.
- **충족기준**:
  - 충분함: 서로 성격이 다른(같은 부서 변형이 아닌) 적용처 2개 이상, 각각 최소한의 방법 설명 동반.
  - 부족함: 적용처 1개(게이트 통과선), 또는 여러 개지만 사실상 동일 작업의 반복 나열.
  - 없음: 적용처 자체가 없음(→ 카테고리 하한).
- **write-post 연결점**: `🌍 다른 업무에 적용한다면?` + Q8. **이미 커버됨(가점축)** — 게이트는 1개, 이 항목은 만점 변별용.

### 8. 확장성(팀 규모/조직 유형 일반화)
- **항목명**: 1인→팀→조직으로 스케일되는가
- **왜 채점자가 이걸 볼 것 같은지**: 개인 생산성 트릭과 조직에 이식 가능한 워크플로우는 적용 가치가 다르다. 채점자는 "혼자만 되는 것"인지 "팀/다른 규모 조직에도 얹을 수 있는가"를 살핀다.
- **충족기준**:
  - 충분함: 다른 팀 규모나 조직 형태(스타트업/대기업/1인)에 적용할 때의 방식·조정점이 최소 1문장 언급.
  - 부족함: 개인 사례에 머무르나 확장 여지는 암시.
  - 없음: 확장성에 대한 고려 전무.
- **write-post 연결점**: `🌍 다른 업무에 적용한다면?`. **미커버 — 선택적 심화**(게이트화하면 인터뷰 과부하이므로 비게이트 권장).

### 9. 전제조건·제약의 정직한 명시
- **항목명**: 적용에 필요한 도구/데이터/권한/비용 전제의 공개
- **왜 채점자가 이걸 볼 것 같은지**: "따라 하면 됩니다"가 진짜인지 판단하려면 숨은 전제(유료 툴, 특정 데이터 접근, 사내 권한)가 드러나야 한다. 채점자는 전제를 감춘 과장된 적용 가능성을 낮게 본다.
- **충족기준**:
  - 충분함: 재현에 필요한 최소 전제(툴/구독/데이터/권한 등)가 1개 이상 명시돼 독자가 자기 상황 적용 여부를 판단 가능.
  - 부족함: 전제가 부분적으로만 드러남.
  - 없음: 전제 언급이 전혀 없어 재현 가능성 검증 불가.
- **write-post 연결점**: `🌍 다른 업무에 적용한다면?` / `📋 재사용 프롬프트`. **미커버 — 선택적 심화**.

### 10. 시나리오의 진정성(anti-fabrication 정합)
- **항목명**: 적용 시나리오가 근거 있는 실체인가, 채우기용 지어내기인가
- **왜 채점자가 이걸 볼 것 같은지**: 25점을 노려 그럴듯한 cross-domain 예시를 **날조**하면 오히려 신뢰가 무너진다. 채점자(및 게이트)는 실제 경험/논리에 뿌리박은 시나리오와 템플릿-스터핑을 구별한다. 정직한 "해당 없음"이 허구보다 낫다.
- **충족기준**:
  - 충분함: 시나리오가 본문의 실제 경험·방법에서 자연스럽게 도출되고, 검증 가능한 구체가 있음.
  - 부족함: 그럴듯하나 본문 근거와 연결이 약한 추상적 확장.
  - 없음/감점: 근거 없는 수치·사례·도메인을 지어냄(→ 게이트의 3단계 예외처리로 "해당 없음" 표기가 정답).
- **write-post 연결점**: 자가진단 게이트(anti-fabrication 3단계) 전반. **이미 커버됨(무결성 방어축).**

### 11. 한계·비적용 영역의 인지
- **항목명**: "이건 안 되는/조심할 경우"의 언급
- **왜 채점자가 이걸 볼 것 같은지**: 적용 가능성을 균형 있게 제시하려면 만능론이 아니라 경계가 있어야 한다. "이런 업무엔 부적합/이 부분은 사람이 검수" 같은 경계 설정은 오히려 방법의 신뢰도와 실전 적용성을 높인다.
- **충족기준**:
  - 충분함: 이 방식이 잘 안 맞는 상황이나 반드시 사람이 개입할 지점을 1개 이상 언급.
  - 부족함: 한계 언급이 형식적(한 단어)이거나 뻔한 면피성.
  - 없음: 무제한 만능처럼만 서술.
- **write-post 연결점**: `🌍 다른 업무에 적용한다면?` / `💬 배운 AI 활용 팁`. **미커버 — 선택적 심화**(가점용, 비게이트).

## (A) 필수 확인 질문 / HARD 게이트 후보

> 25점 배점이므로 20점 카테고리보다 한 항목 더 허용하되, **가장 레버리지 큰 3~4개**로 제한(인터뷰 과부하 방지). 아래는 모두 기존 Phase-3 goksung 질문/게이트에 이미 정렬돼 있어 신규 부담이 거의 없다.

1. **[게이트 유지] 진짜 cross-domain 시나리오 ≥1** (항목 1)
   - 근거: 카테고리의 정의 자체. 현행 HARD 게이트 본체.
   - 판정: 다른 직무/도메인 + 방법 + 왜 되는지. 없으면 Q8 1회 질문 → 그래도 없으면 정직한 "해당 없음".
2. **[게이트 유지] 복붙 가능한 재사용 프롬프트/템플릿 ≥2** (항목 2)
   - 근거: '독자가 지금 가져갈 물건'의 유무가 적용 가능성의 물적 토대. 현행 게이트(재사용 프롬프트 ≥2)와 동일.
   - 판정: 코드/인용블록 형태 ≥2 + 각 1줄 설명. 미달 시 Q7 1회 질문.
3. **[게이트 정합] 즉시 실행 가능한 구체성 — "막연함 불가"** (항목 4)
   - 근거: 게이트가 이미 "vague 불가"를 요구. 항목 1·2가 형식만 채우고 실행 해상도가 낮으면 반려하는 품질 조건.
   - 판정: 시나리오·프롬프트가 '다음 행동'을 특정 가능한 수준인지.
4. **[무결성 게이트] 시나리오·프롬프트의 진정성(anti-fabrication)** (항목 10)
   - 근거: 25점을 노린 날조 방지. "해당 없음"이 허구보다 우선.
   - 판정: 근거 없는 도메인/수치/사례가 출력에 없을 것(게이트 3단계 예외처리로 강제).

> 신규로 게이트를 늘리지 않음 — A그룹은 전부 기존 게이트/질문(Q7·Q8·자가진단)으로 충족. 미세 개선 1건만 선택 권장: 항목 3(파라미터화)을 Q7 프롬프트 요청 시 "`[대괄호]`로 치환 지점 표시"로 문구 보강(게이트화 X).

## (B) 선택적 심화 확인 (비게이트 — 만점 변별·품질 가점용)

- **항목 3** 재사용 산출물 파라미터화 — Q7 문구 미세 보강으로 유도(권장, 저비용).
- **항목 5** 비개발자 실행 장벽 — 본문 소스코드 게이트와 정합, 팁 섹션에서 자연 유도.
- **항목 6** ROI 일반화 가능성 — "이 절감이 남에게도 재현될 이유?" 1회 질문(선택).
- **항목 7** 적용 폭(복수 도메인) — 게이트는 1개, 2개 이상이면 만점 방향 가점.
- **항목 8** 확장성(팀/조직 규모) — 비게이트, 여유 있을 때 1문장 유도.
- **항목 9** 전제조건·제약 명시 — 비게이트, 신뢰도 가점.
- **항목 11** 한계·비적용 영역 — 비게이트, 균형감 가점.

> 설계 요약: 25점 카테고리를 **HARD 게이트 4축(1·2·4·10)**으로 하한을 보장하고, **선택 7축(3·5·6·7·8·9·11)**으로 25점 만점을 향한 변별을 확보한다. 신규 인터뷰 질문은 0개(기존 Q7·Q8 재사용), 미세 문구 보강 1건만 권장하여 인터뷰 부담을 늘리지 않는다.
