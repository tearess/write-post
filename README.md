# write-post

AI 코딩 도구로 만든 것을 **사례글로 남기고**, 채점 루브릭에 맞춰 **최적화**하는 Claude Code 스킬입니다.

- **기본 모드**: AI 활용 사례글 자동 작성 (DEVLOG → 게시글)
- **goksung 모드**: 100점 채점 루브릭 기준으로 최적화된 사례글 생성

---

## 🎯 특징

### 기본 모드 (`/write-post`)
- DEVLOG 자동 생성 (5개 AI 도구 지원)
- 비개발자 대상 사례글 작성
- 한 번에 하나씩 질문하는 저마찰 인터뷰
- 복붙 가능한 재사용 프롬프트 포함

### goksung 모드 (`/write-post goksung`)
- **100점 루브릭 기준 최적화**:
  - 문제정의 (20점)
  - 해결과정 (20점)
  - 결과물 완성도 (25점)
  - 업무 적용 가능성 (25점)
  - 사례 공유력 (10점)

- **신규 인터뷰 질문 3개**:
  1. 정량적 심각도 ("숫자로 표현하면?")
  2. 기존 대안 검토 ("왜 기존 방법은 안 됐나?")
  3. **AI 결과 검증** ("AI 출력을 어떻게 확인했나?") ⭐ 핵심

- **9항목 자가진단 게이트**:
  - 동작 증거 (스크린샷/링크/로그)
  - 정량 before/after 지표
  - 재사용 프롬프트 ≥2개
  - cross-domain 적용 시나리오
  - 에피소드 구조 (요청→작업→결과→배운점)
  - 소스코드 미노출
  - 기존 대안 부재 명시
  - 민감정보 마스킹 ✓
  - 정직한 한계 표기

- **anti-fabrication 3단계**:
  - 근거 없는 항목은 절대 지어내지 않음
  - 실제 답변에서만 추출
  - 없으면 "해당 없음"으로 정직하게 표기

### 수동 설치

`.claude/skills/write-post/` 폴더에 이 저장소의 `.claude/skills/write-post/SKILL.md` 복사:

```bash

cp <write-post-repo>/.claude/skills/write-post/SKILL.md .claude/skills/write-post/
```

설치 후 **Claude Code 도구를 재시작**해야 스킬이 인식됩니다.

## 🚀 사용법

### 기본 모드 (기존 write-post)

```bash
/write-post
```

**프로세스**:
1. **Phase 1**: DEVLOG 생성 (세션 파일 파싱)
2. **Phase 2**: DEVLOG 확인 및 수정
3. **Phase 3**: 사례글 작성 (한 번에 하나씩 질문)

**결과**: `AI_CASE_STUDY.md`

### goksung 모드 (채점 루브릭 최적화)

```bash
/write-post goksung
```

**트리거**: 요청에 리터럴 **"goksung"** 포함 (대소문자 무시)
- ✅ `/write-post goksung`
- ✅ `write-post 스킬로 goksung 모드 실행`
- ❌ `/write-post` (기본 모드)
- ❌ `/write-post 채점` ("채점"만으로는 미발동)

**프로세스**:
1. Phase 1~2: 기본 모드와 동일
2. Phase 3: **goksung 후속질문 추가**
   - 정량 심각도, 기존 대안 검토, AI 검증 방법, 도구 선택 이유, cross-domain 시나리오 등
3. Phase 5: **자가진단 게이트** (9항목 체크)
   - 미달 항목은 1회 재질문 후 "해당 없음"으로 정직히 표기
   - 지어내기 절대 금지

**결과**: `AI_CASE_STUDY.md` (루브릭 최적화)

**선택사항**: 채점 대응표 포함 (요청 시에만)
- 글 끝에 루브릭 카테고리 → 섹션 매핑 테이블 추가 가능

## 📋 100점 루브릭 구조

### 1️⃣ 문제정의 (20점) — 12개 세부 항목

| 항목 | 설명 |
|------|------|
| 구체성 | 특정 업무/도구/반복작업 명시 |
| 정량적 심각도 | 시간/비용/빈도 수치 포함 |
| 시작 계기 | 특정 사건/시점/감정 기술 |
| 긴급성 | 미루면 누적되는 손실 명시 |
| 범위 | 개인/팀/조직 영향 범위 |
| 독자 관련성 | 타깃 독자와 문제 정합 |
| 문제-해결 논리 | Before pain ↔ 솔루션 기능 1:1 대응 |
| 생생함 | 구체 장면/감정 묘사 |
| **기존 대안 검토** ⭐ | **왜 기존 방법은 부족했나** (goksung 신규) |
| 반복성 | 빈도 신호 (매일/매주/매번) |
| 성과 측정 | Before/After 단위 대응 |
| 독자 구체성 | 페르소나 수준의 구체 |

### 2️⃣ 해결과정 (20점) — 11개 세부 항목

| 항목 | 설명 |
|------|------|
| 접근법 논리 | 문제→방법 인과 연결 |
| **방법/도구 선택 근거** ⭐ | **다른 방법/도구 대비 왜 이것** (goksung 신규) |
| 시행착오 구체성 | 특정 실패 + 원인 + 개선 |
| 의사결정 근거 | 판단 기준 드러남 |
| AI 능동 협업 | 사용자의 개입·판단·수정 장면 |
| 반복/개선 사이클 | 1차→발견→2차 개선 순환 |
| 난이도 대비 우아함 | 기술 난이도 대비 간결함 |
| 재현 가능성 | 원문 코드블록 ≥2개 |
| 막힘 원인 정직성 | AI 오해 vs 내 요청 미흡 구분 |
| 인간-AI 역할 분담 | 사람/AI 몫의 명확한 구분 |
| 에피소드 인과 | 요청→행동→결과→배운점 4단 |

### 3️⃣ 결과물 완성도 (25점, HARD) — 10개 세부 항목

| 항목 | 설명 |
|------|------|
| **동작 증거** | 스크린샷/링크/로그 ≥1 |
| **정량 Before/After** | 수치 쌍 (3시간→10분 등) |
| 엔드투엔드 완결성 | 진입→핵심동작→결과획득 완결 |
| 예외 처리 | 실패/예외 상황 대응 언급 |
| 시각 증거 품질 | 캡션/맥락과 함께 배치 |
| 객관적 검증성 | 제3자가 확인 가능한 경로 제공 |
| **재사용 프롬프트** | 복붙 가능 프롬프트 ≥2개 |
| 지속 재현성 | 반복 사용 가능 형태 |
| **정직한 한계** | 완성/미완 경계 명확 + "해당 없음" 표기 ⭐ |
| 결과-목표 정합성 | 서두 목표 = After 지표 대응 |

### 4️⃣ 업무 적용 가능성 (25점, HARD) — 11개 세부 항목

| 항목 | 설명 |

| 한계·비적용 | 부적합 상황/사람 개입 지점 |

## 📝 goksung 모드 상세 가이드

### 1단계: 트리거 확인

요청에 리터럴 **"goksung"** 포함 필수:
```bash
# ✅ 작동
/write-post goksung
/write-post goksung 모드로
write-post 스킬 goksung 옵션

# ❌ 작동 안 함
/write-post
/write-post 채점
/write-post 루브릭
```

### 2단계: 신규 인터뷰 질문

기본 8개 질문 + **goksung 신규 3개**:

| 구간 | 신규 질문 | 목적 |
|------|----------|------|
| 문제상황 | "불편함을 숫자로?" / "기존 대안은?" | 문제정의 20점 강화 |
| 작업하이라이트 | "방법/도구를 고른 이유?" | 해결과정 20점 강화 |
| 결과임팩트 | "AI 결과를 어떻게 확인?" ⭐ | 결과물 25점 핵심 |

각 질문은 "한 문장, 없으면 스킵" 형태 — 저마찰 원칙 유지.

### 3단계: 자가진단 게이트 (9항목)

초안 작성 후 **Phase 5에서 자동 실행**:

```
[ ] 1. 정량 before/after 수치 있는가
[ ] 2. 스크린샷/링크/로그 증거 있는가
[ ] 3. 재사용 프롬프트 2개 이상인가
[ ] 4. Cross-domain 시나리오 있는가
[ ] 5. 에피소드 2개 이상 (각 원문 코드블록)
[ ] 6. 소스코드 노출 없는가
[ ] 7. 기존 대안 검토 담겼는가
[ ] 8. 민감정보(회사명/API 키) 마스킹됐는가
[ ] 9. 미완/한계 정직하게 표기됐는가
```

**미달 항목 처리**:
1. DEVLOG + 사용자 답변에서 실제 근거 확인
2. 근거 없으면 **1회만** 추가 질문
3. 그래도 없으면 **절대 지어내지 말고** "해당 없음"으로 표기

### 4단계: 채점 대응표 (선택)

사용자가 요청하면 글 끝에 **채점용 매핑 테이블** 추가:

```markdown
## 📎 채점 대응표 (제출용 — 커뮤니티 공유 시 삭제)

| 루브릭 카테고리 | 관련 섹션 | 세부 항목 |
|---|---|---|
| 문제정의 (20) | 📝 한줄요약 + 😫 문제상황 | [12개 세부] |
| 해결과정 (20) | 🔧 작업과정 | [11개 세부] |
| ...
```
